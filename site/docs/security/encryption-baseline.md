# Encryption baseline

## Что шифруем

- Чувствительные поля с пользовательскими/финансовыми данными, которые сохраняются в БД в рамках бизнес-модулей.
- Значения, которые нельзя хранить в открытом виде из-за требований безопасности и комплаенса.

## Что не шифруем

- Технические идентификаторы и служебные поля (`id`, внешние ключи, флаги статусов, даты бизнес-событий), если они не содержат секрет.
- Данные, необходимые для фильтрации/агрегации на уровне БД в открытом виде, когда они не относятся к чувствительным.
- Поля, уже защищённые иным формализованным механизмом (например, токенизация во внешнем контуре).

## Где будет жить master key

- Master key хранится вне кода и вне БД: в защищённом секрет-хранилище окружения (например, Vault/KMS/переменные среды рантайма).
- Приложение получает доступ к ключам через абстракцию `SecretKeyProviderInterface` без прямой привязки к конкретному провайдеру.

## Почему не используем plaintext в БД

- Plaintext в БД повышает риск утечки при компрометации дампов, резервных копий или доступов к SQL.
- Шифрование на уровне приложения уменьшает blast radius и позволяет управлять ротацией ключей.
- Единый контракт шифрования обеспечивает повторное использование и консистентность подхода между модулями.

## Naming contract для колонок БД

Для зашифрованного логического поля `<field>` используется строгий набор колонок:

- `<field>_encrypted`
- `<field>_key_version`
- `<field>_encrypted_at`

Контракт зафиксирован в `App\Shared\Security\Contract\EncryptionFieldNamingContract`.
