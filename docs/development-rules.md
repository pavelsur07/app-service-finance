# Правила разработки и развития проекта

Цель: развивать проект без хаоса, сохраняя предсказуемость и стабильность.

## 1) Модульный принцип
- Новая функциональность добавляется только через модуль: `src/<ModuleName>/...`
- Общие папки `src/Service`, `src/Controller`, `src/Entity`, `src/Repository`, `src/Form` считаются legacy/shared.

## 2) Запрет на “свалку”
- Нельзя добавлять новый код в legacy/shared-папки без отдельного решения и причины.
- Нельзя добавлять новые шаблоны в `templates/_partials`.

## 3) Правило PR
- 1 PR = 1 атомарное изменение.
- В PR обязательно указать: что изменено, где проверено.

## 4) Правило тестов
- Всегда должен проходить “зелёный коридор”: `composer test:smoke`.
- Падающие исторические тесты допускается помечать `@group legacy`.
- Для прогонов без legacy использовать: `composer test:no-legacy`.

## 5) Правило документации
- Любой новый документ обязан быть добавлен ссылкой в `docs/README.md`.
- Любая фича/изменение поведения требует обновления документации.
