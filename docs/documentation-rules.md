# Правила ведения документации

Цель: чтобы документация не превращалась в “свалку”, а оставалась пригодной для разработки и поддержки.

## 1) Формат и структура
- Документация пишется в Markdown.
- Один документ = одна тема.
- Любой новый документ обязан быть добавлен ссылкой в `docs/README.md`.

## 2) Куда писать
- Запуск/команды/рутинные операции: `README.md` или `docker/cron/README.md` или runbook-документы.
- Архитектура и ключевые принципы: документы в `docs/`.
- Интеграции: документы в `docs/` (раздел “Интеграции” в `docs/README.md`).
- Финансовые отчёты/методология: документы в `docs/` (раздел “Финансовые отчёты и методология” в `docs/README.md`).
- Открытые вопросы/сомнения: отдельные файлы в корне или в `docs/` и обязательно ссылка из `docs/README.md`.

## 3) Обновление документации
- Если PR меняет поведение системы, в PR должна быть правка соответствующего документа.
- Если добавляется новый модуль/фича — добавить/обновить документ и ссылку в `docs/README.md`.

## 4) Запрещено
- Нельзя хранить большие регламенты в `README.md` (корень). Там только быстрый старт и команды.
- Нельзя создавать документы без добавления ссылки в `docs/README.md`.
